# 실제 게시 워크플로 (semi-autonomous)

목표: 홍보 에이전트가 **실제로 SNS에 글+미디어를 올린다**. 단, 안전 경계 안에서 — 사람이 최종 발행 버튼 앞에 선다.

## 경계 (바꿀 수 없는 것 — 안전 규칙)
에이전트가 **할 수 없는** 것 (사장님이 직접):
- 계정 로그인 / 비밀번호·토큰 입력 / 계정 생성
- CAPTCHA·봇탐지 통과
- 승인 없는 자동 발행 (발행은 건별 승인)

에이전트가 **할 수 있는** 것 (셋업 후):
- 승인된 카피+미디어를 compose 창에 채우기
- 미디어 자산 파일 준비 (GIF/이미지/영상)
- 건별 승인 하에 발행 버튼 클릭, 또는 사장님이 클릭
- 발행 후 posted_url을 승인 큐에 기록

## 셋업 (사장님이 1회)
1. 게시할 브라우저(Chrome)에 대상 계정 로그인 (X, LinkedIn, Reddit, HN 등).
2. Claude의 Chrome 확장(claude-in-chrome) 연결 확인 — 이게 있어야 에이전트가 브라우저를 구동.
3. 그 후엔 "이 초안 X에 올려줘" 하면 에이전트가 로그인된 세션에서 compose를 채운다.

## 흐름 (초안 → 게시)
```
marketing 초안 (approval_queue, status: draft)
      │
사장님 검토 → status: approved
      │
"올려줘" 지시 → 에이전트가 브라우저 compose 채움 (카피 + 미디어 첨부)
      │
[건별 승인] "지금 발행? y/n" → 발행 클릭
      │
posted_url 기록 → status: posted  (metrics가 반응 추적)
```

## 플랫폼별 메모
- **HN**: 공식 게시 API 없음 → 브라우저 경로만. Show HN은 news.ycombinator.com/submit.
- **X**: 브라우저 compose. 미디어(GIF/영상) 업로드 = 로그인 세션에서.
- **LinkedIn**: 브라우저 compose. 개인 계정 명의.
- **Reddit**: 서브레딧별 규칙·자동화 민감 → 반드시 사람 눈으로 최종 확인. 크로스포스트 하루 간격.
- **API 경로(선택)**: X/LinkedIn/Reddit는 공식 API+토큰도 가능하나, 앱 등록·토큰 발급은 사장님이. 발급되면 API 레인으로 전환 가능(브라우저보다 안정적).

## 원칙
- 첫 런치의 첫 게시들은 **반드시 사람이 최종 클릭** — 바이럴 순간에 사람이 지켜보는 게 정석.
- 완전 무인 자동발행은 하지 않는다(정책 + 플랫폼 밴 리스크 + 평판 리스크).
